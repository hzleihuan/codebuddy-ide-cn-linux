## 新增模块分析与移植处理

### 1. `@tencent/qimei-node@1.2.2`

- **背景介绍**：
  `@tencent/qimei-node` 是腾讯内部私有 Node.js SDK，对应移动端大家熟知的 QIMEI（腾讯设备可信标识 / 风控设备指纹体系）的服务端 Node 版，仅腾讯内部业务、游戏、MSDK 体系可访问，公网 npm 无公开包，外部开发者无法直接安装使用。
  *   **QIMEI**：腾讯灯塔体系下的设备识别风控组件，全称设备可信标识，核心用于设备指纹、黑产识别、异常设备判定、数据上报归因。
  *   **qimei-node**：服务端 Node 语言封装包，专门在后端做 QIMEI 相关校验、生成、上报、解密。
  *   **`@tencent/`**：腾讯内部 scope 私有包，托管在内网 npm（如 tnpm），外部无分发。
- **类型**：腾讯专有的私有 Node 模块（未发布在 npm 公共仓库）。
- **平台支持**：仅包含 Windows（`src/win/`）和 macOS（`src/mac/`，依赖 `QimeiSDKMac.xcframework`）相关的底层实现，没有 Linux 支持代码。
- **运行分析**：在 CodeBuddy 主进程代码（`out/main.js`）的初始化过程中，该模块的初始化函数 `initSdk()` 在进入具体逻辑前，首行便有平台校验：
  ```javascript
  if (process.platform !== "darwin" && process.platform !== "win32") {
      this.available = false;
      return;
  }
  ```
  这意味着在 Linux 平台上运行时，该服务会**直接提前返回**，完全不加载任何底层的平台 `.node` 模块。
- **移植处理**：
  在 `scripts/lib/native-modules.sh` 的 `remove_known_wrong_platform_modules()` 中新增清理逻辑，直接删除该模块下的编译目录、mac 目录和 win 目录，只保留 JS 接口文件防错，不引入任何多余体积与二进制校验警报。
  ```bash
  rm -rf "$app_dir/node_modules/@tencent/qimei-node/build" 2>/dev/null || true
  rm -rf "$app_dir/node_modules/@tencent/qimei-node/src/mac" 2>/dev/null || true
  rm -rf "$app_dir/node_modules/@tencent/qimei-node/src/win" 2>/dev/null || true
  ```

### 2. `koffi@2.16.2`

- **类型**：快速 C/C++ FFI（外部函数接口）模块，上游引入它是作为 Windows 平台加载 `@tencent/qimei-node` 动态库时的 FFI 依赖。
- **运行分析**：因为 `@tencent/qimei-node` 在 Linux 上直接退出不加载，因此作为其下级依赖的 `koffi` 在 Linux 上也**根本不会被加载或运行**。
- **移植处理**：
  `koffi` 带有跨平台的多平台预编译二进制目录（`build/koffi/`）。为了避免编译并且为了满足潜在的 `require` 引用依赖，我们选择保留该模块目录，但为了优化体积，在 `remove_known_wrong_platform_modules()` 中**只保留 `linux_x64` 和 `linux_arm64` 的原生文件**，将 Windows、macOS、FreeBSD、OpenBSD 等无关架构/系统的预编译二进制全部裁撤：
  ```bash
  if [ -d "$app_dir/node_modules/koffi/build/koffi" ]; then
      find "$app_dir/node_modules/koffi/build/koffi" -mindepth 1 -maxdepth 1 \
          ! -name "linux_x64" ! -name "linux_arm64" -exec rm -rf {} + 2>/dev/null || true
  fi
  ```
