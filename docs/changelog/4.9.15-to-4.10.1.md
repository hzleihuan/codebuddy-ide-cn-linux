| 版本 | 4.9.15 | 4.10.1 | 变化 |

## Node 模块版本对比与处理方式

由于本版本将数据源彻底从 **macOS DMG 提取** 切换为 **官方原生 Linux DEB 解包**，我们对原生模块的重构策略进行了重大调整：
1. **直接复用原生 Linux 二进制**：对 DEB 中已自带 Linux `x86_64` `.node` 原生二进制 of 模块，直接予以保留，免去不必要的重构与网络下载。
2. **缺失模块本地补译**：对官方 DEB 未打入 Linux 预编译二进制的必备模块，保留原 DMG 移植工作流的自动构建逻辑，通过 `@electron/rebuild` 针对 `37.7.0` 头文件在本地进行按需编译。
3. **平台垃圾清理**：在 `install.sh` 阶段对 Windows 专有和 macOS 专有的模块和二进制文件进行清理。

## 🛠️ 移植流程的核心代码变动

### 1. 本地动态补译策略实现 (`scripts/lib/native-modules.sh`)

在 `rebuild_native_modules` 阶段，为了避免重复编译已带有原生 Linux `.node` 文件的模块，我们在循环中引入了自动探针：
```bash
# 检查是否已包含 native 二进制，若包含则跳过编译
local has_node
has_node=$(find "$app_dir/node_modules/$module_name" -name "*.node" 2>/dev/null | head -1)
if [ -n "$has_node" ]; then
    info "Module $module_name already has precompiled binary ($has_node), skipping rebuild"
    continue
fi
```
该机制保证了：
* **`@parcel/watcher`、`spdlog`、`sqlite3` 等模块** 100% 自动跳过，无需耗时编译，且避开了在低配环境下因原生依赖不足可能导致的编译报错。
* **`node-pty`、`native-keymap`、`native-watchdog` 等模块** 在检测到缺失后，仍然会正常下载并用宿主系统的环境将其完美 rebuild，确保了 Linux 终端等基础功能的完美可用。

### 2. 构建依赖精简化 (`scripts/install-deps.sh`)
由于解包替代了复杂的重新编译 and 跨平台镜像转换，开发机的依赖树大量精简，但为了满足 `node-pty` 编译以及高清图标生成的需要，我们保留并精简了基础开发库：
* 保留：`binutils`（提供 `ar` 提取 deb）、`tar`、`desktop-file-utils`、`ImageMagick`（用于在 `install.sh` 阶段将官方 `150x150` 的非标图标自动放大为 `256x256` 标准高清图标）。

## ⚠️ 移植过程中遇到的关键问题与解决方案

### 1. RPM 打包负载压缩格式错误 (`w3.gztst`)
* **问题现象**：在 Fedora 上执行 `make package` (RPM) 时，打包工具链报错：
  ```
  错误：未知的净负荷压缩：w3.gztst
  ```
* **原因分析**：在优化构建性能引入 `_binary_payload` 自定义压缩以加速大体积 Electron 目录打包时，使用了非标准的压缩宏缩写 `w3.gztst`。
* **解决方案**：在 `scripts/build-rpm.sh` 中将其修正为 RPM 官方支持的标准快速 gzip 压缩格式 `%define _binary_payload w3.gzdio`。

### 2. 官方 DEB 缺失 `node-pty` 导致内置终端无响应
* **问题现象**：完成移植解包后，IDE 可以成功启动并显示窗口，但在新建终端时，右下角弹出警告：`与终端的pty主机进程的连接无响应，终端可能会停止工作。`
* **原因分析**：官方发布的 Linux DEB 存在打包漏洞，没有将底层 C++ 编写的 `pty.node` 原生动态链接库打入包中，仅打包了 JS 包装代码，导致 node-pty 初始化失败。
* **解决方案**：
  * 恢复编译工具链（`nodejs`, `npm`, `python3`, `make`, `gcc-c++` 以及键盘布局相关的 `libX11-devel`, `libxkbfile-devel` 依赖库）。
  * 改造 `scripts/lib/native-modules.sh`，在解包阶段检测如果对应模块（如 `node-pty`, `native-keymap`, `native-watchdog`）缺失 Linux 原生 `.node` 二进制文件，则自动联网从 npm 拉取源码并针对宿主的 Electron `37.7.0` 运行时环境执行 `@electron/rebuild` 重新编译，最后回塞到安装包中。

### 3. AppImage 启动后产生 `SIGBUS (Signal 7)` 总线错误崩溃
* **问题现象**：在宿主桌面上运行构建的 AppImage 时，主界面偶尔闪退，查看 `journalctl` 或系统 coredump 日志发现：
  ```
  Command Line: /tmp/.mount_xxxx/usr/bin/codebuddycn-app/buddycn --no-sandbox
  Signal: 7 (BUS)
  ```
* **原因分析**：
  * 原先的 AppImage 的入口 `AppRun` 执行的是脚本包装器 `start.sh`，这会间接执行官方的 CLI 脚本 `bin/buddycn`。
  * `bin/buddycn` CLI 在启动时以脱离后台进程（detached）的方式拉起真实的 `buddycn` GUI 窗口进程，然后 CLI 包装器进程本身随之立刻成功退出。
  * AppImage 运行时一旦检测到 `AppRun` 链路上的父进程退出，便会立即强行卸载（umount）位于 `/tmp/.mount_xxxx` 的 FUSE 挂载点。
  * 此时刚启动 of GUI 主窗口进程还在加载，当它去读取已被卸载的挂载路径中的代码或内存映射文件（如 V8 snapshot）时，便会因为物理文件突然消失而触发操作系统的 `SIGBUS` 段错误崩溃。
* **解决方案**：
  在 `scripts/lib/appimage.sh` 中修改 AppImage 的 `AppRun` 生成模板，使其跳过 CLI 包装脚本，**直接前台执行 `buddycn` Electron 可执行二进制文件**：
  ```bash
  exec "$SELF_DIR/usr/bin/codebuddycn-app/buddycn" --no-sandbox "$@"
  ```
  这样，真正的 GUI 主进程就作为 AppImage 的首要前台进程运行，阻止了 FUSE 挂载点被提早卸载，彻底解决了 `SIGBUS` 崩溃问题。

### 4. 切换语言等触发的应用内自动重启 (Relaunch) 闪退
* **问题现象**：在 AppImage 中尝试更换显示语言（如切换至简体中文）时，界面提示需要重启应用以生效。但在点击“重新启动”按钮后，程序直接退出且未能重新打开。
* **原因分析**：
  * 当用户在应用内选择重启时，Electron 会调用 `app.relaunch()` 来启动新进程。
  * `app.relaunch()` 默认总是执行当前的 `process.execPath`。在 AppImage 挂载状态下，其物理路径指向了内部临时解压的 `/tmp/.mount_xxxx/usr/bin/codebuddycn-app/buddycn`。
  * 这新进程会直接越过 AppImage 外层的 `AppRun` 壳脚本，因此也失去了在 `AppRun` 中硬编码添加的 `--no-sandbox` 命令行参数。
  * 在不支持无特权命名空间或对沙箱有限制的系统环境下（如 Fedora 44），新进程在启动时初始化沙箱失败，导致 Chromium 自毁闪退。
  * 试图通过覆写 `process.execPath` 为外部 `.AppImage` 路径来解决此重启问题会产生严重的副作用，这会导致 VS Code 内部通过执行 `process.execPath` 来进行 Shell 环境解析（unable to resolve your shell environment）的 helper 子进程全部崩溃失效（退码 9）。
* **规避与解决办法**：
  * 这是 Electron 框架在 AppImage 环境下的已知打包限制。
  * 当用户更改语言或由于设置更改需要重启应用时，**只需在程序退出闪退后，手动重新点击启动一次 AppImage 即可**。语言和配置在此前已经被正常写入配置文件中，二次手动运行会完全正常应用。
